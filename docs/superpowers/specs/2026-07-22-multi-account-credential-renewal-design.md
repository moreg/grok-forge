# 技术文档

## 1. 需求主导人

莫宇生

## 2. 需求描述

### 2.1 背景

桌面端已经具备多账号展示、token 保存和账号选择能力，但账号选择目前只更新前端元数据，没有将所选账号的凭据绑定到 Grok Agent。Grok Agent 本身已经支持 OIDC `refresh_token`、启动时静默刷新、到期前主动刷新、请求前刷新和 401 恢复刷新，但认证状态以单一 `auth.json` 为中心。

本需求需要同时满足以下目标：

- 当前账号在应用运行期间自动续期，尽量避免登录失效。
- 长期未使用的账号在再次切换或打开其历史任务时按需静默恢复。
- 不修改 xAI/OIDC 服务端策略。
- 应用关闭后不保留后台进程，也不注册系统计划任务。
- 支持标准浏览器 OIDC 登录和导入凭据；只有完整、可信的 OIDC 凭据可以自动续期。
- 不支持或改造 API Key 登录。
- 不允许不同账号之间串用凭据、任务或 ACP 会话。

客户端不能承诺账号永不失效。若服务端因闲置有效期、绝对有效期、用户撤销、密码或安全策略变化而拒绝 `refresh_token`，目标账号必须重新登录。

### 2.2 设计决策

采用“每个账号独立 `GROK_AUTH_PATH`，同一时间运行一个账号对应的 Agent”方案。

不采用以下方案：

- 不将 `AuthManager` 改造成单进程多账号容器，避免认证上下文传播范围过大和账号串用风险。
- 不在切换时覆盖全局 `~/.grok/auth.json`，避免轮换型 refresh token 出现双份副本、并发覆盖或重用检测。
- 不通过后台常驻进程保活非当前账号。

### 2.3 总体架构

```text
桌面端账号注册表（不含密钥）
        │
        ├── accountId / 名称 / 来源 / 续期能力 / 最近使用时间
        │
        ▼
AgentProcessManager
        │ 根据 accountId 解析固定认证路径
        │ 启动 Agent 时设置 GROK_AUTH_PATH
        ▼
应用数据目录/accounts/{accountId}/auth.json
        │
        ▼
现有 AuthManager
  ├── 启动时静默刷新
  ├── 到期前主动刷新
  ├── 请求前刷新
  └── 401 后恢复刷新
```

组件职责如下：

- `AccountRegistry`：保存账号名称、来源、续期能力、最近使用时间和最后一次已知状态，不保存任何密钥。
- `AccountCredentialStore`：在 Tauri/Rust 侧管理每个账号独立的认证文件，负责导入识别、校验、检查和删除。
- `AgentProcessManager`：记录 Agent 当前绑定的账号，执行受控切换，并阻止前端传入任意认证文件路径。
- `TaskAccountBinding`：将任务和创建任务时的账号绑定，防止跨账号恢复 ACP 会话。
- 现有 `AuthManager`：继续保持单账号职责，复用其刷新、文件锁、原子写入和失败退避逻辑。

### 2.4 数据模型

账号非敏感元数据：

```ts
type Account = {
  id: string
  name: string
  source: 'browser-oidc' | 'import'
  renewal: 'refreshable' | 'non-refreshable' | 'unknown'
  authStatus:
    | 'unknown'
    | 'valid'
    | 'refreshing'
    | 'temporarily-unavailable'
    | 'relogin-required'
  createdAt: number
  lastUsedAt: number
}
```

任务模型增加：

```ts
type Task = {
  accountId: string | null
  // 现有字段保持不变
}
```

`authStatus` 是最后一次已知状态，不是长期可信事实。应用启动、账号切换或请求前必须重新检查。旧任务迁移为 `accountId = null`，首次打开时由用户选择归属账号。

### 2.5 凭据分类与续期规则

#### 标准浏览器 OIDC 登录

通过 Agent 初始化响应中的 `authMethods` 选择 OIDC 方法，并调用 ACP `authenticate`。登录成功后，完整凭据写入当前账号的认证文件。应用运行期间继续使用现有主动刷新机制。

不再将“打开登录网页并要求用户手工粘贴 token”当作浏览器登录。

#### 完整 OIDC 导入包

自动续期至少需要以下字段：

```json
{
  "access_token": "...",
  "refresh_token": "...",
  "expires_at": "...",
  "issuer": "...",
  "client_id": "..."
}
```

Tauri 后端负责字段校验、到期时间解析、issuer HTTPS 校验和允许域名校验，再转换为现有 `AuthStore/GrokAuth` 格式。前端只接收脱敏后的识别结果。

#### 浏览器凭据、单一 token 或 cookie

- 只识别明确的白名单字段，不通过模糊字符串猜测 refresh token。
- 只有能够可靠取得完整 OIDC 刷新参数时才标记为 `refreshable`。
- 缺少任何必要刷新参数时降级为 `non-refreshable`，界面必须明确提示过期后需要重新导入或改用标准 OIDC 登录。
- 不允许用户通过导入内容指定任意刷新地址。

### 2.6 生命周期

```text
登录或导入成功
  → 原子写入账号 auth.json
  → 正常使用
  → 到期前约 5 分钟刷新
  → 服务端可能轮换 refresh token
  → 新凭据在同一文件锁保护下原子写回
```

- 应用启动：仅启动当前账号 Agent；已过期的 access token 立即尝试静默刷新。
- 应用运行：当前账号按现有策略主动续期。
- 切换账号：停止旧 Agent，使用目标账号认证路径启动新 Agent，并立即检查或刷新。
- 非当前账号：不定时唤醒，不因 `lastUsedAt` 太久被本地删除或判定失效。
- 应用关闭：不执行任何刷新任务。
- 长期未使用后再次选择：refresh token 有效时无感恢复；被服务端拒绝时进入 `relogin-required`。
- 用户主动退出：删除当前账号认证文件，不影响其他账号。

### 2.7 账号切换流程

```text
当前账号
  → 检查运行中任务
  → 有任务则弹窗确认
  → 取消并短暂等待当前任务
  → 停止旧 Agent
  → 使用目标账号 GROK_AUTH_PATH 启动 Agent
  → ACP 初始化并检查凭据
  → 可用 / 静默续期 / 要求重新登录
```

- 同一时间只允许一个切换操作；重复操作合并或拒绝。
- 存在运行中任务时，弹窗列明切换将中断的任务。确认后先发送取消请求，再停止 Agent。
- 目标账号需要重新登录时，账号切换仍成功，但进入目标账号专属登录状态。
- Agent 启动、IPC 或 ACP 初始化等技术故障发生时，回滚 `currentAccountId` 并尝试恢复原账号 Agent。
- 网络或服务临时异常标记为 `temporarily-unavailable`，保留凭据并允许重试。
- `invalid_grant`、明确撤销等永久错误标记为 `relogin-required`，停止自动重试，防止刷新风暴。

### 2.8 任务隔离

- 创建任务时写入当前 `accountId`。
- 打开其他账号任务时弹窗确认；确认后先切换账号，再加载任务。
- 前端和 Agent 启动层都校验任务账号，禁止 A 账号的 ACP 会话交给 B 账号恢复。
- 导入任务时不信任外部 `accountId`；仅当本机存在对应账号时才能绑定，否则保持未归属。
- 删除账号后历史任务默认保留，并统一转成未归属状态。

### 2.9 删除账号

- 删除前展示关联任务数量。
- 删除当前账号时先停止 Agent，再要求用户选择其他账号。
- 删除认证文件成功后再删除账号元数据。
- 凭据删除失败时保留账号元数据并提示错误，避免界面显示已删除但磁盘仍残留密钥。
- 删除账号不会自动删除历史任务。

### 2.10 安全要求

- 原始导入内容只在 Tauri/Rust 后端处理，不进入 React 状态、`localStorage`、日志、遥测或错误详情。
- 前端不得传入认证文件绝对路径；后端只接受系统生成且格式合法的 `accountId`。
- 账号认证目录必须解析在固定应用数据目录以内，防止路径穿越。
- 认证文件沿用现有安全文件权限、文件锁和原子写入能力。
- issuer 必须是 HTTPS，并匹配内置允许域名或管理员配置。
- 导入前只展示账号标识、到期时间和续期能力，不回显任何密钥。
- 导入成功后清理输入缓冲，并删除迁移完成的旧钥匙串副本。
- “能够解析”不代表“凭据有效”；首次启用仍须由服务端验证。

### 2.11 兼容迁移

- 现有全局 `~/.grok/auth.json` 不复制，避免产生同一轮换 refresh token 的双份副本。
- 首次升级时，由 Tauri 后端创建一个带内部迁移标记的“默认旧账号”，继续直接使用原路径。只有这个由后端生成并记录的迁移账号可以解析到全局认证文件；普通 `accountId` 始终解析到应用数据目录，前端不能通过伪造 ID 选择任意路径。
- 现有钥匙串账号在首次选择时执行一次导入识别；成功后迁移到独立认证文件并删除旧钥匙串项。
- 迁移失败时保留原始项并提示用户，不静默丢失凭据。
- 旧任务保持未归属，首次打开时由用户绑定。

### 2.12 验收标准

- 两个 OIDC 账号分别登录后，认证文件互相独立，切换账号不会覆盖另一账号凭据。
- 当前账号在 access token 到期前能够复用现有机制自动续期。
- 关闭应用一段时间后再次打开，refresh token 有效的当前账号能够静默恢复。
- 长期未使用账号再次切换时，refresh token 有效则无感恢复；永久失效则只要求该账号重新登录。
- 账号切换不会把原账号任务或 ACP 会话加载到目标账号。
- 网络临时故障不会清除凭据或错误地要求重新登录。
- 完整可信的 OIDC 导入包标记为可续期；不完整 token/cookie 明确标记为不可续期。
- 删除一个账号不会删除其他账号凭据或历史任务。
- 应用关闭后没有凭据刷新后台进程或系统计划任务。

## 3. 影响范围

### 3.1 桌面前端

- `apps/desktop/src/lib/accounts.ts`：账号模型、状态和迁移元数据。
- `apps/desktop/src/App.tsx`：当前账号、切换协调和任务账号绑定。
- `apps/desktop/src/components/Overlay.tsx`：登录、导入、账号状态、切换和删除交互。
- `apps/desktop/src/lib/grokAcpClient.ts`：解析 `authMethods`，调用 ACP `authenticate`。
- `apps/desktop/src/lib/tasks.ts`：持久化及导入任务的 `accountId`。
- `apps/desktop/src/lib/desktopBridge.ts`：新增账号凭据和进程管理桥接接口。

### 3.2 Tauri 后端

- `apps/desktop/src-tauri/src/lib.rs`：账号路径解析、凭据导入检查、删除、Agent 账号绑定及 `GROK_AUTH_PATH` 注入。
- Tauri capability/schema：随新增命令同步更新。

### 3.3 Grok Shell

原则上复用现有 `AuthManager`，不改造成多账号管理器。若实现中发现 ACP 初始化未完整暴露认证状态，只允许增加最小的脱敏状态输出，不改变 refresh token 核心协议。

### 3.4 不在范围内

- 修改 xAI/OIDC 服务端有效期或撤销策略。
- API Key 登录和 API Key 生命周期。
- 应用关闭后的后台保活、Windows 服务或系统计划任务。
- 多账号 Agent 并行运行。
- 自动删除历史任务。

## 4. 测试要点

根据项目规范，本需求不主动新增单元测试类、集成测试类或测试方法，也不主动启动项目进行运行验证。实现完成后必须执行并通过以下编译检查：

- 桌面前端 TypeScript 构建检查。
- Tauri Rust 编译检查。
- 涉及 Grok Shell 时执行对应 Rust crate 编译检查。

需要重点核对的行为包括：

- 账号路径解析不能逃逸应用数据目录。
- 账号切换时环境变量绑定到目标认证文件。
- 轮换后的 refresh token 只存在于目标账号认证文件。
- 技术故障回滚账号，永久认证失败不回滚账号。
- 跨账号打开任务必须先确认和完成账号切换。
- 迁移旧全局认证文件时不复制 refresh token。
- 导入凭据的错误和日志均不包含原始密钥。

## 8. 其它

### 8.1 已知限制

- 不运行后台进程意味着应用关闭期间无法定期刷新；再次启动或切换账号时才尝试恢复。
- refresh token 最终有效期由服务端决定，客户端无法保证永久免登录。
- 普通 token/cookie 在缺少完整刷新参数时不能升级为可续期凭据。

### 8.2 实施顺序建议

1. 增加账号安全路径解析和 `GROK_AUTH_PATH` 注入。
2. 为任务增加账号归属和跨账号打开保护。
3. 接入 ACP 标准浏览器 OIDC 登录。
4. 增加凭据导入、识别和脱敏状态接口。
5. 完成旧全局认证及旧钥匙串账号迁移。
6. 补齐账号状态、删除和故障回滚交互。
